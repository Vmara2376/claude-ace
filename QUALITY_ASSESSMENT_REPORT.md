# ACE-Coder 质量优先评估报告

> "Token 节省是手段，代码质量才是目的。"

在追求极致的 Token 优化时，我们必须回答一个核心问题：**这些优化是否以牺牲代码生成质量为代价？**

为了回答这个问题，我设计了一组严谨的 A/B 测试，对比了“全量读取（基线）”与“骨架模式（ACE）”在不同类型任务下的代码生成正确性。

## 1. 实验设计与结果

我们模拟了一个真实的业务模块（包含配置解析、数据转换、重试逻辑等），并设计了三个不同复杂度的任务。所有生成的代码都必须通过严格的单元测试。

| 任务类型 | 任务描述 | 全量模式 (基线) | 骨架模式 (ACE) | 质量影响 |
|---|---|---|---|---|
| **T1: 精确定位型** | 修改 `retryWithBackoff` 函数，增加 jitter 参数 | ✅ PASS (1015 tokens) | ✅ PASS (570 tokens) | 🟢 **无损失** (省 43.8%) |
| **T2: 跨函数依赖型** | 重构 `parseConfig` 支持嵌套配置 | ✅ PASS (1103 tokens) | ✅ PASS (1157 tokens) | 🟢 **无损失** |
| **T3: 全局理解型** | 添加 `batchProcess` 函数，复用现有逻辑 | ✅ PASS (1087 tokens) | ❌ **FAIL** (614 tokens) | 🔴 **质量下降** |

### 失败原因深度剖析 (T3)
在 T3 任务中，骨架模式生成的代码在测试中抛出了 `Cannot read properties of undefined (reading 'prefix')`。
**为什么？** 因为在骨架模式下，AI 看不到 `parseConfig` 函数的内部实现，它不知道 `parseConfig` 返回的对象结构中包含 `prefix` 字段。因此，它在编写 `batchProcess` 时，错误地假设了配置对象的结构，导致运行时崩溃。

---

## 2. 核心结论：优化的边界在哪里？

实验数据清晰地划定了各项优化的“安全边界”：

### 🟢 纯收益优化（无质量风险）
1. **意图驱动的验证闭环 (`IntentVerificationTool`)**
   - **结论**：这是**提升**代码质量的最强手段。通过强制 TDD 和自愈循环，它不仅没有风险，反而能兜底其他优化带来的潜在错误。
2. **批判性架构师 (`CriticalArchitectTool`)**
   - **结论**：纯收益。在代码生成前拦截糟糕的设计，从源头保证质量。
3. **跨项目长期记忆 (`CrossProjectMemory`)**
   - **结论**：纯收益。提供额外的上下文，不破坏现有逻辑。

### 🟡 有条件收益优化（需谨慎使用）
1. **骨架模式 / 语义图谱 (`SemanticSearchTool` & `ContextLoader`)**
   - **结论**：这是一把双刃剑。
   - **安全区**：用于“寻找代码位置”、“修改独立函数”、“分析类结构”。
   - **危险区**：用于“编写需要深度复用现有逻辑的新功能”。如果 AI 看不到被调用函数的内部实现，极易产生幻觉（如 T3 实验所示）。

---

## 3. 改进方案：如何做到“既要又要”？

为了在保证 100% 代码质量的前提下最大化 Token 节省，我们需要对 ACE-Coder 的 `ContextLoader` 进行**策略降级（Fallback Strategy）**：

### 改进后的自适应策略 (v0.2.0 规划)

1. **默认骨架，按需展开 (Lazy Loading)**
   - 初始状态下，Agent 仍然只看到骨架。
   - **关键改进**：赋予 Agent 一个新工具 `ExpandSymbolTool`。当 Agent 决定复用某个函数（如 `parseConfig`）时，它**必须**先调用该工具展开该函数的完整实现，然后再生成代码。

2. **验证闭环兜底 (Verification Fallback)**
   - 如果 Agent 因为上下文不足生成了错误代码（如 T3），`IntentVerificationTool` 的测试会失败。
   - **关键改进**：在自愈循环中，如果测试失败，系统应自动将相关文件的上下文策略从“骨架”降级为“全量”，再让 Agent 重试。

## 总结

我们不能为了省 Token 而让 AI 变成“瞎子”。真正的理想态，不是盲目压缩上下文，而是**赋予 AI 动态获取所需上下文的能力，并用严格的测试闭环来兜底**。这才是通向高质量 AI 编程助手的正确道路。
